## 一、Superpowers 技能：哪些发挥了最大作用，哪些"形式大于实质"

### 真正有效的技能

**brainstorming（设计探讨）**：这是整个项目中价值最高的技能。它强制在写任何代码之前完成需求澄清、方案对比和设计审批。在我的项目里，逐一追问语言选择、架构模式、重点维度这些问题，迫使我对每一个设计决策有明确的理由。尤其是"一次只问一个问题 + 多选优先"的模式——它没有让我被一连串问题淹没，而是在每个节点上停下来认真思考。最终产出的 SPEC（503 行）覆盖了需求文档的所有要点，没有 TBD 或模糊段落。

**subagent-driven-development（子智能体驱动开发）**：这个技能的执行纪律——每个 task 一个新鲜子智能体、task 间做两阶段审查（规范合规 + 代码质量）、审查不通过不进入下一个 task——确实保证了代码质量。16 个 task 中，Task 5、6、9 都是在审查环节发现了实现者的问题，而不只是实现者自己宣称"做完了"。如果没有强制审查，ToolRegistry 里的 POSIX 兼容性问题和 AgentLoop 的批量工具调用丢失 bug 会留到很晚才发现。

**test-driven-development（测试驱动开发）**：TDD 在 "规范包含了完整代码" 的 task 上价值有限——实现者照着规范抄代码，测试也照着规范抄，红-绿-重构变成了形式。但在没有完整代码模板的 task 上（如 Task 9 主循环、Task 11 WebSocket），TDD 确实驱动了更清晰的接口设计。

### 形式大于实质的技能

**writing-plans（撰写计划）**：计划文档（2754 行）在实现阶段被引用得很少。子智能体只读 task-brief 提取的片段，从不会回到完整的 PLAN.md。计划的价值更多在于"把设计转化为可执行清单"的思维过程，而非作为实现阶段的参考文档。

**verification-before-completion（完成前验证）**：这个技能的要求——先跑验证命令再宣称完成——在实际操作中被子智能体的一致性覆盖了。每个 task 的测试运行是 TDD 流程的一部分，不需要额外提醒。

## 二、TDD 强制在 AI 协作下是阻碍还是放大器

**放大器的场景**：TDD 在接口不明确时迫使你先想清楚"这段代码应该怎么被调用"。写 AgentLoop 的集成测试时，我需要先定义 MockLLM 返回什么响应序列、期望最终是什么状态——这反过来塑造了 `run()` 方法的签名和 `Session` 模型的设计。没有测试先行，我可能会写出一个更难 mock、更难验证的接口。

**阻碍的场景**：当规范里的代码已经写得很完整时，TDD 的"先写失败测试→实现→重构"三步变成了机械表演——测试代码和实现代码都来自同一份规范，测试基本不会在第一次运行失败（除了 import 错误）。这种情况下，TDD 增加了一个循环但没有增加洞察。

**我的判断**：TDD 在 AI 协作下仍然是放大器，但前提是规范是"需求描述"而非"代码清单"。当 AI 帮你写了规范里的完整代码时，TDD 的价值转移到了集成测试层——验证这些组件拼在一起是否正确，而不是单个函数的单元测试。

## 三、Subagent 驱动工作流的能力边界

**自主运行距离**：在 task 描述足够清晰（"实现 X，包含 Y 测试，提交到 Z"）且不依赖跨 task 判断时，子智能体能自主运行完成整个 task 而不偏离主题。我观察到的最长自主运行是 Task 11（FastAPI 服务器）——子智能体创建了 8 个文件，写了 7 个集成测试，全程没有问问题，输出质量达到审查通过标准。

**偏离场景**：Task 9（主循环）是唯一需要升级模型（haiku→sonnet）的 task。最初的实现有两个严重 bug：批量工具调用在重新进入 PLANNING 时被覆盖，AWAITING_HUMAN 状态没有恢复路径。这不是子智能体"偏离"了规范——它忠实地实现了规范——而是规范本身在复杂交互上有设计缺陷。

**最优 task 粒度**：以我的经验，最优粒度是"一个可独立验证的组件 + 它的专属测试"。Task 5（工具注册表 + 5 个工具）应该拆成两个 task——工具注册表本身和具体工具实现是可以分开验证的。Task 9（主循环）的粒度正好——它是一个需要独立判断的完整集成点。

## 四、SPEC/PLAN 质量如何影响实现质量

**具体案例**：Task 5 的规范里写了 `subprocess.run(command, shell=False)` 用字符串而非列表。实现者忠实地实现了这个代码，但审查者指出这在 POSIX 系统上会失败（shell=False 要求命令是一个列表）。这不是实现者的错误——规范本身的代码有 cross-platform bug。修复后（`shlex.split(command)`），规范-实现-审查三者才对齐。

**规律**：规范越接近"完整代码"，实现者越可能照抄代码而不思考上下文。"不要写代码，写需求"这个原则在 AI 协作下更值得遵守——当规范写的是"执行 shell 命令，使用 subprocess 并确保跨平台安全"，实现者会比看到完整代码时更认真地考虑边界条件。

## 五、最有效的 Prompt/Context 策略

**拆分 task 上下文**：每个子智能体只收到三样东西——task brief（规范里的那一段）、接口说明（这个 task 消费和产生的具体函数签名）、全局约束（如 `shell=False`、`Python 3.12+`）。不传任何对话历史，不传前面的 task 做了什么。这比"把整个 PLAN.md 喂给子智能体"效率高得多——子智能体不会对不相关的上下文做错误推理。

**审查提示词的结构**：告诉审查者"不要相信实现者的报告——验证 diff"这个指令显著提高了审查质量。Task 5 的审查者发现了实现者报告里没有提到的 `pytest-json-report` 依赖缺失。相比之下，"请审查代码质量"这种模糊指令会导致审查流于表面。

**失败模式**：当 task brief 只有 47 行（Task 10 管理器）而实际需要实现三个独立组件时，我需要在 dispatch 中补充大量 context。这说明 PLAN.md 中 summary 级别的 task 描述不足以支撑子智能体自主执行。

## 六、凭据与分发：这迫使你想清楚了哪些原本会忽略的问题

**凭据**：最不直观的发现是"桌面和 Docker 需要不同的存储策略"。桌面端钥匙串（Windows Credential Manager）是最佳选择，但 Docker 容器内没有钥匙串服务。这迫使设计了一个自动检测运行环境并选择后端（keyring vs AES-GCM 加密文件）的策略。另外，`mask()` 函数（`sk-...ab12`）虽然简单，但在日志、API 响应和 UI 显示三个层面都要一致使用——如果有一处遗漏，密钥就可能泄露。

**分发**：Dockerfile 的多阶段构建（Node 构建前端 → Python 运行后端）看起来是标准做法，但实施时遇到了两个细节：Vite 的 `outDir` 必须和 Dockerfile 里的 `COPY --from=frontend` 路径一致；`HARNESS_KEY_PASSWORD` 环境变量的默认值（空字符串）会导致没有设置密码的用户遇到模糊的 RuntimeError。这些问题在"只写代码不考虑分发"的项目里根本不会被想到。

## 七、如果重做我会改变什么

1. **SPEC 里不写完整代码**：规范应该描述接口和约束，把实现留给子智能体。这会让 TDD 更有意义，也会让审查者发现更多"规范设计"层面而非"实现偏离"层面的问题。

2. **先做集成测试框架**：Task 9（主循环）和 Task 11（WebSocket）之间有一个空白——缺少一个从 WebSocket 到 AgentLoop 的端到端集成测试。如果在实现前端之前先写这个测试，整体集成会更平滑。

3. **护栏测试覆盖命令白名单**：审查者指出了第 2 层没有单元测试。虽然白名单逻辑很简单，但没有测试意味着未来的修改没有安全网。

4. **更早引入 CI**：`.gitlab-ci.yml` 是最后才创建的。如果一开始就有 CI 在每次推送时运行测试，Task 5 在 POSIX 环境下的 shell=False bug 会更早暴露。

5. **为 SPEC_PROCESS.md 和 AGENT_LOG.md 预留同步更新的时间**：这些过程文档如果等到了最后才补写，会丢失很多过程中的细节判断。

## 八、对 Superpowers 方法论的批判

### Superpowers 做了什么假设？

1. **人类会认真参与**：brainstorming 技能假设用户会仔细回答每一个追问，并在每个设计章节签字确认。如果用户敷衍回答（"都行""随便"），整个流程就会生产一份同样敷衍的 SPEC。

2. **子智能体具有基础工程判断力**：subagent-driven-development 假设子智能体在完成任务的同时能做好自我审查。但 Task 9 的两个严重 bug 说明，即使使用 sonnet 级别的模型，复杂交互中的设计缺陷仍然需要人类审查才能发现。

3. **TDD 的纪律能被 AI 执行**：不管用不用 AI 工具，"先写测试"都是人类工程纪律而非 AI 能力的问题。Superpowers 通过 skill 的内容和流程强制执行，但这种外部纪律和内部理解之间仍有差距。

### 这些假设在我的项目里成立吗？

**假设 1（人类认真参与）**：基本成立。在每个设计决策节点上我确实认真思考了（语言选择、架构模式、护栏设计），但在一些"看起来很简单"的配置选择上我没有深思——比如单用户模型、Web 框架——这些选择在实现阶段没有被质疑，但不代表它们就是最优的。

**假设 2（子智能体判断力）**：部分成立。在"规范有完整代码"的 task 上，子智能体表现完美。在"规范描述需求但不给实现"的 task 上，子智能体需要更明确的接口定义。Task 9（主循环）是两个严重 bug 的来源，但这更多反映了规范本身的设计缺陷而非子智能体的无能。

**假设 3（TDD 纪律被 AI 执行）**：表面上成立——每个 task 都有测试，94/94 通过。但真正有价值的测试是集成层的（test_agent_loop.py, test_websocket.py），单元层的很多测试只是"代码镜像"，没有独立发现 bug 的能力。这不是 AI 的问题——人类写单元测试同样会落入这个陷阱。

### Superpowers 最本质的贡献

Superpowers 的真正价值不在于它提供的任何一个单独技能，而在于它把"先设计→再计划→再实现→再审查"的工作流制度化。在 AI 可以快速产出大量代码的世界里，这种制度化比任何一个具体的技术选择都重要。它像是一个外部的"工程超我"——在你忍不住想"这太简单了，直接写代码吧"的时候拉住你。

但这个制度化也有代价：当项目确实很简单时，走过完整的 brainstorming→writing-plans→subagent-driven-development 流程会让人感觉是在"为流程服务而非为项目服务"。判断什么时候可以简化流程、什么时候必须严格遵守——这种元判断能力，可能比严格遵守流程本身更需要练习。
