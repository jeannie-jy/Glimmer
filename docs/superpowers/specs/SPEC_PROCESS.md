## 一、brainstorming 关键节点

### 1.1 智能体追问的关键问题与设计决策

以下按时间顺序记录智能体逐一追问的问题、我的回答，以及每个决策如何影响了最终 SPEC。

| 序号 | 智能体的提问 | 我的回答 | 对设计的影响 |
|------|-------------|---------|-------------|
| 1 | 编程语言选择？ | Python，曾考虑 Python+Java，智能体分析后建议纯 Python | 确定全栈 Python，影响后续所有技术选型 |
| 2 | 重点深入维度？ | 让智能体推荐，接受反馈闭环为主要贡献 | SPEC 中反馈分析器成为第一重点，包含完整的解析器/分类器/重试策略设计 |
| 3 | LLM 供应商策略？ | 多供应商（Anthropic + OpenAI + Mock） | LLM 抽象层设计为可插拔适配器模式 |
| 4 | 用户交互界面形态？ | 成熟 Web 应用 | 引入 FastAPI + WebSocket + React + Open Design，架构增加 Web 层 |
| 5 | Web 框架选择？ | FastAPI + WebSocket + React | 确定前后端分离架构 |
| 6 | 分发形态？ | Docker + PyInstaller 双分发 | 凭据存储需兼容桌面钥匙串和容器环境变量两种模式 |
| 7 | 凭据存储方案？ | 智能体建议 keyring 优先 + 加密文件回退 | CredentialManager 设计为自动检测运行环境的策略 |
| 8 | 单用户还是多用户？ | 单用户 | 无需认证系统，WebSocket 仅绑 localhost |

### 1.2 用户主动提出的修正

| 修正点 | 我提出的问题 | 对设计的改变 |
|--------|------------|------------|
| 缺失反馈传感器 | ToolRegistry 有 run_tests 但缺少反馈解析机制 | 补上 FeedbackAnalyzer 完整设计（§3.5），包含策略分发、结构化提取、多轮修正策略 |
| 正则护栏脆弱性 | 正则匹配可被编码/间接执行绕过 | 接受智能体建议在 SPEC 中记录局限；后进一步要求加沙箱 |
| 要求加沙箱 | 正则不够，需要纵深防御 | 护栏从单层黑名单扩展为三层架构：路径沙箱 + 命令白名单 + 正则黑名单。治理维度从 ★ 升为 ★★★ 重点 |
| 提醒 SPEC_PROCESS.md | 项目要求同步记录过程文档 | 建立本文档，持续记录 |

### 1.3 AI 提出而我采纳的建议

- **纯 Python 而非 Python+Java 混用**：智能体分析了两种语言在 harness 各组件上的适配性后，指出 Java 没有不可替代的优势，混用会增加 IPC 和分发复杂度。采纳。
- **反馈闭环作为第一重点**：智能体分析"面试官更看重哪部分"时，指出反馈闭环最能体现"机制是代码不是提示词"的核心纪律，且演示效果最直观。采纳。
- **状态机架构方案（三选一中的推荐）**：智能体提出三种方案（状态机/管道/事件驱动），推荐状态机，理由是"移除 LLM 后最易于确定性验证"。采纳。
- **keyring + 加密文件回退方案**：智能体分析了桌面端和 Docker 端的不同安全约束后推荐。采纳。
- **三层沙箱架构**：在用户要求加沙箱后，智能体设计了路径沙箱 + 命令白名单 + 正则黑名单的纵深方案，包含进程隔离（shell=False + timeout）。采纳。

### 1.4 我推翻或修正的 AI 建议

- **反馈分析器遗漏**：智能体在架构总览图中画了 FeedbackAnalyzer，但在组件设计章节跳过直接到了护栏。我指出缺失，智能体补上完整设计。
- **单用户模型确认**：智能体最初建议单用户，我要求先查文档确认——文档无明确规定，确认单用户合理后采纳。

## 二、关键迭代对话节选

### 2.1 语言选择迭代

> **AI**：能不能用 python+java？
> **我**：有意思的组合。你设想两者各自承担什么角色？A) Python 主循环, Java 工具执行 B) Java 主循环, Python LLM 层 C) 还没想好
> **我**：C
> **AI**：[分析了两种语言的适配性，结论：纯 Python。Java 没有不可替代的优势，混用增加 IPC 复杂度]
> **我**：那就纯 Python

**决策**：智能体的分析有理有据——每个子系统逐一分析了 Python 和 Java 的适配度，指出护栏的本质是字符串匹配（Python 够用），沙箱在 OS 层（语言无关），分发会引入双运行时负担。被说服。

### 2.2 重点维度选择迭代

> **AI**：项目要求六个维度都有最低实现，需选一个深入。A) 治理 B) 反馈闭环 C) 扩展
> **我**：我都想要有，你觉得哪个重要一点？面试官会更看重哪部分？
> **AI**：推荐 B 反馈闭环。[分析了工程深度和演示效果的对比]
> **我**：可以

**决策**：智能体的论证包含两个角度——技术深度（确定性代码 vs 提示词）和展示价值（闭环演示最直观），有说服力。后续又因我要求加沙箱，治理也升级为重点，最终形成双重点。

### 2.3 反馈分析器补全迭代

> **我**：1. 缺失了反馈传感器...你需要一个机制（校验器）来解析这些产物，客观判定它是成功还是失败。2. 正则护栏的脆弱性...在你的 SPEC 中建议客观指出局限性...
> **AI**：两个点都说得对。[补上 FeedbackAnalyzer 完整设计] [同意护栏局限分析]
> **我**：你能加上沙箱吗
> **AI**：[设计了 PathSandbox + Whitelist + PatternBlacklist 三层架构]

**决策**：这两轮修正显著提升了设计的工程深度——反馈分析器有了策略模式、结构化输出、重试上限；护栏从脆弱单层变成了纵深三层。从结果看，主动质疑是值得的。

## 四、反思：brainstorming 技能的评价

### 做得好的地方

1. **一次只问一个问题**：这确实是有效的实践。逐一追问逼迫我在每个决策点停下来思考，而非被一连串问题淹没后随意作答。
2. **多选优于开放**：大多数问题提供了 2-3 个具名选项，降低了决策成本。尤其是在我不确定时（如 Web 框架选择），选项帮我快速聚焦。
3. **方案对比包含推荐理由**：状态机 vs 管道 vs 事件驱动的对比中，智能体以"可测性"作为核心判准，与项目要求文档的判准一致，论证有针对性而非泛泛而谈。
4. **分块呈现设计并逐块确认**：避免了"读完一整篇 SPEC 发现方向全错"的风险。

### 让我不满的地方

1. **架构图讲到但组件遗漏**：FeedbackAnalyzer 在架构总览图里出现，但在组件描述部分被跳过。如果我没有主动发现，这可能成为 SPEC 的设计缺陷。智能体在分块呈现时应有交叉检查——某一节提到的组件是否在其他节有详细说明。
2. **对安全问题的初始分析偏浅**：正则护栏的脆弱性是我提出的，智能体第一版只做了单层正则匹配。对于自称"重点维度"的治理模块，初始设计深度不足。好在追问和修正机制弥补了这一点。
3. **没有主动提醒 SPEC_PROCESS.md**：这是我事后补充的（根据项目文档要求），智能体在 brainstorming 过程中未提及。如果能在 brainstorming 开始时就提醒"我会在过程中同步生成 SPEC_PROCESS.md"，体验更好。

### 整体判断

brainstorming 技能作为一个"流程脚手架"是有效的——它守住了"先设计再实现"的纪律，追问质量也足够。但它的产出质量高度依赖用户的质询能力——如果我没有追问沙箱和反馈分析器，最终 SPEC 会浅不少。这不是技能的缺陷，而是它设计时就预设了"人类会认真参与"——这个假设在认真做项目的场景下成立。
